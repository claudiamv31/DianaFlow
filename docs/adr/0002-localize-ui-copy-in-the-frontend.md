# Localize UI Copy in the Frontend

DianaFlow will support three to five curated locales with `react-i18next`, initially `en-US` and `es-MX`. English is the canonical source, locale catalogues are bundled with the frontend, only complete and human-reviewed locales may be activated, and the device-local preference resolves before browser language and the `en-US` fallback. CI will enforce catalogue and interpolation parity, plural correctness, backend content-key and error-code coverage, and the absence of hard-coded user-facing text; a development-only pseudo-locale will expose layout and translation gaps.

The backend will return stable lowercase domain codes, machine-readable error codes, and language-neutral Daily Insight and Daily Focus keys instead of localized prose. Guidance keys will be selected deterministically from the user, local calendar date, Cycle Phase Estimate, and content type, so the semantic guidance remains stable across refreshes, server instances, and language changes. Fixed insight keys will live in backend source rather than the database, allowing the `PhaseMessages` table and cache to be removed. Unknown keys and error codes will produce translated safe fallbacks rather than raw internal values.

All displayed dates, numbers, and plurals will use the active locale through `Intl` and `i18next`; locale will not appear in application URLs. Language names will use autonyms, document direction will follow the locale, and all Daily Insight and Daily Focus translations will follow the non-diagnostic Cycle Phase Estimate language defined in the frontend and backend glossaries.
